# SCHEMA.md

Каноничная схема LLM Wiki. Этот файл — первая точка входа для AI-агентов и людей перед любым ingest, query, audit или изменением второго мозга.

Схема **не описывает текущее состояние как идеальное**. Она задаёт целевое устройство Wiki. Если текущие страницы не соответствуют схеме, это backlog качества, а не повод ослаблять схему.

## 1. Назначение Wiki

`Company-LLM-Wiki` — корпоративный второй мозг для:

- накопления проверяемых знаний о бизнесе, продуктах, продажах, финансах, процессах и AI-агентах;
- фиксации управленческих выводов, решений, рисков, гипотез и открытых вопросов;
- подготовки контекста для будущих AI-агентов;
- снижения повторной работы: один раз найденное знание должно быть доступно через ссылки, MOC, dashboards и поиск.

Главный критерий страницы: **она должна помогать понимать, управлять или развивать бизнес**.

## 3. Слои знания

### 3.1 Raw / source layer

- `01_Raw` — неизменяемые сырые источники. AI-агент не редактирует их без явного разрешения.
- `02_Sources/Source Registry.md` — реестр источников: что это, где лежит, статус, как использовать.
- `02_Sources/Source Summaries/` — очищенные summary крупных источников.
- `02_Sources/Extracted Facts/Insight Inbox.md` — быстрый буфер для полезных, но ещё не скомпилированных инсайтов.

### 3.2 Stable wiki layer

`03_Wiki` содержит знания, которые уже скомпилированы в устойчивые страницы: продукт, процесс, метрика, финансы, синтез, решение, стратегия, операция.

Стабильная страница не должна быть просто копией источника. Она должна отвечать: **что известно, откуда известно, что это значит для управления**.

### 3.3 Navigation / management layer

- `04_MOC` — карты содержания по областям.
- `05_Dashboards` — управленческие точки наблюдения, KPI, monitoring specs.
- `index.md` — верхнеуровневая навигация, не полный каталог всех страниц.
- `log.md` — append-only журнал всех значимых операций.

## 5. Frontmatter

Каждая markdown-страница в Wiki, кроме временных черновиков внутри `01_Raw`, должна начинаться с YAML frontmatter.

Обязательные поля:

```yaml
---
type: synthesis | metric | process | product | decision | ...
status: draft | in-review | active | outdated | deprecated | archived
owner: your-org
updated: YYYY-MM-DD
confidence: high | medium | low | unknown
source:
  - source name or path
tags:
  - tag
---
```

Рекомендуемые поля, когда применимо:

```yaml
review_after: YYYY-MM-DD
related:
  - related-page
source_count: 2
decision_date: YYYY-MM-DD
metric_owner: your-org
data_source: PostgreSQL | ClickHouse | Google Sheets | codebase | manual
supersedes:
  - old-page
superseded_by:
  - new-page
```

Правила:

- `updated` менять при каждом содержательном изменении страницы.
- `source` должен указывать источник фактов, не только `current session`, если есть конкретный источник.
- `confidence: high` ставить только при подтверждении документом, БД, кодом или несколькими независимыми источниками.
- `confidence: low` или `unknown` требует явного `to verify` / открытого вопроса в тексте.

## 6. Статусы

- `draft` — полезно, но не финализировано; можно использовать осторожно.
- `in-review` — готовится к закреплению, требуется проверка.
- `active` — актуальная рабочая версия.
- `outdated` — устарело, но ещё полезно как история.
- `deprecated` — заменено новым знанием; должна быть ссылка `superseded_by`.
- `archived` — перенесено в `09_Archive` или больше не участвует в навигации.

## 7. Уровни знания внутри страниц

В тексте явно различать:

- **fact** — подтверждено источником;
- **insight / synthesis** — вывод из фактов;
- **hypothesis** — рабочая гипотеза;
- **decision** — принятое решение;
- **risk** — риск;
- **open question** — вопрос;
- **to verify** — требует проверки.

Нельзя превращать гипотезу в факт без проверки. Нельзя молча удалять противоречащий старый вывод — его нужно пометить как устаревший или перенести в противоречия.

## 8. Маршрутизация знаний

Правило по умолчанию:

| Тип знания | Куда писать |
|---|---|
| Быстрый инсайт, неясная маршрутизация | [[02_Sources/Extracted Facts/Insight Inbox]] |
| Новый источник | [[02_Sources/Source Registry]] + summary при необходимости |
| Подтверждённый факт | Релевантная страница `03_Wiki` |
| Метрика / модель данных | `03_Wiki/Metrics` |
| Финансовый источник или ОПиУ | `03_Wiki/Finance` + `Source Registry` |
| Продажи / дневной разбор | `03_Wiki/Sales` |
| Управленческий вывод | `03_Wiki/Synthesis` |
| Процесс / правило работы | `03_Wiki/Processes` |
| Продуктовый факт | `03_Wiki/Products` |
| Решение | `03_Wiki/Decisions/Decision Log.md` + decision page при необходимости |
| Противоречие | `03_Wiki/Decisions/Contradiction Register.md` |
| Управленческий сигнал для наблюдения | `05_Dashboards` |
| Работа агента | `06_Agents/Agent Reports` |

Если агент не уверен — писать в Insight Inbox с пометкой `to route`, но не терять знание.

## 9. Правила создания и обновления страниц

Создавать новую стабильную страницу, если:

- это самостоятельный продукт, процесс, метрика, решение, источник, dashboard или сквозной синтез;
- тема повторяется в нескольких источниках;
- вопрос будет регулярно всплывать в будущих задачах;
- текущая страница разрастается больше ~200 строк и требует разделения.

Не создавать страницу, если:

- это одноразовая мелкая деталь;
- информация не относится к вашему бизнесу/business-development;
- это PII, секрет, токен, доступ или необезличенная персональная информация;
- достаточно записи в Insight Inbox.

При обновлении:

1. Прочитать `SCHEMA.md`, `index.md`, `log.md` и релевантный MOC/страницу.
2. Проверить поиском, нет ли существующей страницы/дубля.
3. Обновить содержание и `updated`.
4. Добавить/проверить wikilinks.
5. Обновить MOC/dashboard/index, если изменилась навигация или управленческий обзор.
6. Добавить запись в `log.md`.
7. Закоммитить и запушить изменения Wiki, если они внесены агентом и пользователь не запретил push.

## 10. Wikilinks и навигация

### 10.1 Иерархия навигации

Канон навигации: **`index.md` → MOC/registry → частные страницы**.

- `index.md` — только верхнеуровневые точки входа: главные MOC, dashboards, schema/rules, ключевые индексы. Он не должен становиться полным каталогом частных кейсов.
- `04_MOC/*` — тематические карты содержания: продукты, процессы, метрики, продажи, финансы, агенты, решения.
- Доменные registry/index pages — место для списков частных кейсов: помесячные ОПиУ, ежедневные/недельные разборы продаж, cohort checks, agent reports, source summaries.
- Частные страницы — конкретный месяц, день, неделя, проверка, кейс, выгрузка, отчёт агента. Они должны ссылаться вверх на свой registry/MOC и на 1–3 действительно релевантные стабильные страницы.

Примеры правильной маршрутизации:

- Разбор P&L за май → отдельная страница `03_Wiki/Finance/P&L 2026-05`; список таких страниц — `03_Wiki/Finance/P&L - Помесячный разбор (индекс)`; из `index.md` ссылка только на этот индекс.
- Проверка конверсии за конкретную дату/период → отдельная страница в `03_Wiki/Sales` или `03_Wiki/Metrics`; список — доменный индекс вроде `<Продукт X> - Ежедневный разбор (индекс)` или метрика/воронка-index; из `index.md` ссылка только на MOC/registry.
- Agent report → отдельный файл/папка в `06_Agents/Agent Reports`; список — `Agent Reports Registry`; из `index.md` ссылка только на `MOC - AI Agents` / registry при необходимости.

### 10.2 Правила ссылок и защита от link sprawl

- На стабильных страницах `03_Wiki`, MOC, dashboards и agent workflows должны быть meaningful wikilinks на связанные страницы.
- Цель — не формальная плотность ссылок, а возможность пройти по графу знаний.
- Страницы без исходящих ссылок допустимы только как временный backlog, шаблон, raw/source artifact или техническая справка; для стабильных business-pages это дефект качества.
- При создании новой важной страницы добавить её в релевантный MOC/registry; в `index.md` добавлять только верхнеуровневые точки входа.
- Не делать «полную перелинковку со всем»: частная страница обычно имеет 2–5 исходящих ссылок, а не 15–30.
- Не добавлять обратную ссылку на каждую частную страницу из каждой связанной страницы. Обратная навигация идёт через registry/MOC, а не через ручные backlinks везде.

### 10.3 Обратные навигационные ссылки и циклы

Допустимы короткие **обратные навигационные ссылки** вида `MOC/registry → частная страница → тот же MOC/registry`. Это аналог «оглавление → глава → вернуться к оглавлению» и не является содержательной зависимостью.

Не формулировать это как общее разрешение на циклы. Плохой цикл — когда страницы начинают зависеть друг от друга по содержанию или создают маршрут без ясного выхода: `case → dashboard → synthesis → case → ещё 20 ссылок`.

Запрещённые/нежелательные паттерны:

- `index.md` ссылается на десятки частных кейсов напрямую;
- каждый кейс вручную добавляется в несколько MOC и dashboards;
- stable synthesis page начинает хранить полный список дневных/месячных разборов;
- dashboard превращается в архив всех фактов;
- agent report остаётся единственным местом важного знания без компиляции в `03_Wiki`.

Хороший паттерн:

```text
index.md
  → MOC - Metrics / MOC - Sales / MOC - AI Agents
    → доменный индекс или registry
      → частная страница кейса
        → вверх: registry/MOC
        → вбок: 1–3 стабильные страницы (метрика, процесс, synthesis)
```

### 10.6 Анти-галлюцинационная дисциплина

Главное правило: **не утверждать то, чего нет в источнике**.

1. **Нет числа без источника.** Каждое ключевое число имеет провенанс через `source` в frontmatter или через `derived_from`/`evidence` в тексте.
2. **Неуверенное помечается явно.** Вывод без прямого источника помечается `(inferred)`; спорная интерпретация — `(ambiguous)`.
3. **Спорное → Contradiction Register.** Если новая информация противоречит зафиксированной, сначала пишем в [[03_Wiki/Decisions/Contradiction Register]], не перетираем.
4. **Нет данных — пишем «не найдено».** Не подставлять «скорее всего», «типично», «обычно» без источника. Лучше `to verify` или `unknown`, чем убедительная выдумка.
5. **Confidence честный.** `confidence: high` только при подтверждении документом, БД, кодом или несколькими независимыми источниками.

> **Откатные практики.** Поле `claim:` в frontmatter и под-документные узлы `entity::` тестировались как расширение метода. Два изолирующих A/B-эксперимента (по 90 решателей-субагентов, слепой судья, фиксированные golden-ответы) показали: они не улучшают точность retrieval, а entity-узлы работают как аттракторы — уводят агента на страницу сущности в обход канонической страницы-метрики. Обе практики из метода убраны. Числа должны жить ровно в одном месте — на странице-источнике или метрике — с провенансом через `derived_from::`/`evidence::`.

### 10.4 Периодические разборы: datestamped листы

Частный случай §10.1–10.3 для повторяющихся периодических разборов (ОПиУ по месяцам, воронка/конверсия по месяцам, дневные/недельные разборы продаж). Добавляет к общим правилам только конкретику оформления:

- **Индекс-реестр** — `type: moc`; держит сводную таблицу по периодам + метод чтения + ссылки вниз на листы; вешается на профильный MOC.
- **Лист** — именовать `<Тема> YYYY-MM` (или конкретная дата); один период; ссылка вверх на свой индекс-реестр.
- **Листы одного ряда НЕ ссылаются друг на друга** (усиление anti-sprawl §10.2): кросс-периодные сравнения (тренды месяц-к-месяцу) держать в индексе-реестре, а не в листах. Иначе N периодов дают до N² ручных связей.
- Разросшийся индекс-реестр (>200 строк / перегруженный хаб, см. §9, §17) — вынести аналитику в `03_Wiki/Synthesis`, в реестре оставить таблицу периодов и ссылки.

Примеры в работе: `03_Wiki/Finance/P&L - Помесячный разбор (индекс)`, `03_Wiki/Metrics/<Продукт X> - Воронка (помесячно)`, `03_Wiki/Metrics/<Продукт X> - Конверсия когорт (помесячно)`.

## 11. Источники и provenance

- Не сохранять устойчивые утверждения без источника.
- Для БД-аналитики фиксировать: источник (`PostgreSQL`, `ClickHouse`, Google Sheets), период, фильтры, grain, исключения.
- Для кода фиксировать путь к файлу/классу/методу, но не копировать большие фрагменты кода без необходимости.
- Для Google Sheets фиксировать fileId, лист, диапазон/колонки, дату доступа.
- Raw PII не сохранять. Агентов/клиентов обезличивать, использовать агрегаты и технические ID только если они нужны для воспроизводимости.

## 12. Противоречия и устаревание

Если новая информация противоречит старой:

1. Не перетирать старое утверждение молча.
2. Указать дату, источник и область применимости обеих версий.
3. Если противоречие существенно — добавить запись в [[03_Wiki/Decisions/Contradiction Register]].
4. Если старое знание superseded — поставить `status: deprecated` или добавить `superseded_by`.
5. Если нужен контроль — добавить `review_after`.

## 13. Decisions

Решение — это не просто инсайт. Решение должно иметь:

- дату;
- кто/что является источником решения;
- контекст;
- принятое действие;
- последствия/риски;
- дату пересмотра (`review_after`) для обратимых или спорных решений.

Решения фиксируются в [[03_Wiki/Decisions/Decision Log]] и, если решение крупное, отдельной page в `03_Wiki/Decisions`.

## 14. Dashboards

Dashboard — не архив фактов, а управленческий экран. Он должен содержать:

- что наблюдаем;
- почему это важно;
- источники данных;
- частоту обновления;
- пороги/алерты или критерии внимания;
- ссылки на метрики, процессы, synthesis и последние разборы.

Разовые выводы не пихать в dashboard без необходимости; сначала synthesis или Sales/Finance page, затем dashboard только если сигнал повторяемый.

## 15. Agent reports

`06_Agents/Agent Reports` хранит результаты делегированной работы и сложных исследований.

Отчёт агента должен иметь:

- frontmatter;
- задачу;
- метод;
- источники;
- ключевые выводы;
- ограничения;
- что нужно скомпилировать в `03_Wiki`.

После отчёта важные выводы нужно перенести в стабильные Wiki-страницы. Agent report не должен быть единственным местом, где живёт важное знание.

## 16. Git-политика для Wiki

Для второго мозга действует синхронизация через Git:

- изменения Wiki, внесённые агентом, должны быть закоммичены и запушены без отдельного запроса, если пользователь не запретил push;
- не коммитить не относящиеся к Wiki временные SQL, выгрузки, credentials, `.env`, raw PII и локальный мусор;
- перед commit: `git status`, `git diff --cached --name-status`, `git diff --cached --check`;
- staged scope должен включать только изменения текущей Wiki-задачи;
- commit message писать по смыслу, например `docs(wiki): add wiki schema`.

## 17. Lint / health-check

Регулярный audit должен проверять:

1. битые wikilinks;
2. отсутствующий frontmatter;
3. отсутствующие `owner`, `status`, `source`, `confidence`;
4. страницы без meaningful outbound links;
5. duplicate basenames, которые ломают Obsidian-навигацию;
6. decisions без `review_after`;
7. low-confidence страницы без `to verify`;
8. draft pages, которые давно не обновлялись;
9. source registry gaps;
10. важные знания, застрявшие в Insight Inbox;
11. страницы >200 строк, которые требуют split;
12. PII/secrets в staged diff.

При audit отличать:

- **P0:** broken links, missing frontmatter, secrets/PII, неверная навигация важного источника;
- **P1:** стабильные business-pages без ссылок/источников, decisions без review date, knowledge trapped in Inbox;
- **P2:** draft/low-confidence backlog, page-size, style cleanup.
