# brain-name-clusters

**Purpose:** Make the vault's knowledge topology visible and navigable. The diagnostics script detects CNM communities (groups of densely connected pages); this skill names them so humans and agents can navigate by cluster.

Inspired by graphify's Leiden communities with LLM-named labels (e.g. "OSAGO KV & Margin Economics"). Our CNM communities are equivalent; they just need the naming pass.

---

## When to invoke

- After a major ingest that adds 10+ new pages
- During periodic brain-review
- When onboarding someone new to the vault (cluster names orient them quickly)
- When the graph feels navigable but not legible

---

## Steps

### 1. Run graph diagnostics

```bash
brain diagnose
# or directly:
python3 scripts/graph_diagnostics.py --vault /path/to/vault
```

Collect the output section labeled **communities** or **CNM clusters**. Each community is a list of page names grouped by topological density.

### 2. Read each community

For each community:
- Read the page titles
- Skim 2–3 pages' `claim` fields to understand the domain
- Identify the unifying theme

### 3. Name each community

Naming rules:
- One concise phrase (3–7 words)
- Domain + function: "Product X Conversion Funnel", "Finance KV Economics", "Agent Delegation Patterns"
- Avoid generic names: "Misc", "Other", "General" → either find the theme or mark as `unclustered`
- Use the domain's own vocabulary, not abstract labels

**Example names (from our vault):**
- "OSAGO Conversion & Funnel Metrics"
- "Finance KV & Margin Economics"
- "Agent Delegation & Role Architecture"
- "Product X Channel Economics"
- "Decision & Risk Register"

### 4. Write the clusters note

Create or update a page in the vault (e.g. `08_Graph/Knowledge Clusters.md` or `04_MOC/MOC - Clusters.md`):

```markdown
---
type: moc
status: active
updated: YYYY-MM-DD
tags:
  - meta
  - graph
---

# Knowledge Clusters

Generated by: `brain diagnose` on YYYY-MM-DD
Vault size: N pages, M edges

## Clusters

### 1. <Cluster Name>

**Core pages:** [[Page A]], [[Page B]], [[Page C]]
**Theme:** <one sentence describing what unifies this cluster>
**Size:** N pages

...
```

### 5. Identify unclustered / bridge pages

The diagnostics output will also show:
- **Orphan pages** (no connections) → candidates for brain-link or brain-review
- **Bridge pages** (connecting two otherwise unrelated clusters) → high-value, worth highlighting
- **Hub pages** (high in-degree) → check if they are overloaded; flag for possible split

### 6. Add cluster note to navigation

- Tag the note with `meta` (keeps it out of the knowledge graph view)
- Reference from the vault's main index or top-level MOC
- Note that cluster names are snapshots — they should be regenerated after major ingests

### 7. Log the naming pass

Operation log entry: date, vault size at time of run, number of communities found, any notable findings (new cluster emerged, a major bridge page identified, orphan count).

---

## Notes on the method

CNM (Clauset-Newman-Moore) communities are detected by `graph_diagnostics.py` using modularity optimization over the typed-edge graph. The communities reflect actual link density, not tags or folders. A page's cluster membership can surprise you — that surprise is often the most valuable output of this skill.

If graphify is available: its Leiden communities (with LLM-generated names already included) can be imported as a second layer. The two community detections will not be identical; comparing them reveals structural insights.
